## Methodology disclosure

The spec calls for "axe-core (vitest) + manual keyboard navigation test". This audit was executed by a CLI agent without browser/screen-reader/keyboard-driver capability. The audit pairs:

1. **Automated axe-core scan** of representative composites for each of the 7 pages (rendered via `@testing-library/react` + jsdom). Tests live in `apps/web/src/__tests__/a11y/*.a11y.test.tsx` and assert `toHaveNoViolations()`. All 7 pass with 0 violations.
2. **Direct read** of layout, header, footer, and key interactive components (vote arrows, Nala panel, status pill, kelas filters, janji filters) for ARIA correctness and landmark coverage.
3. **Static contrast math** against the 11 brand tokens (`apps/web/src/app/globals.css`), comparing computed luminance ratios to WCAG AA thresholds.

**Caveat**: jsdom does not implement `getComputedStyle` for color values, so axe-core's `color-contrast` rule cannot fire reliably and is disabled in the test helper (`shared/render-with-axe.tsx`). Color contrast is audited manually below in Section C — and is the area with the most outstanding work. Manual keyboard navigation (Tab order, focus-visible indicator, Esc dismissal) and screen-reader behaviour (NVDA/VoiceOver) should be confirmed by Mas during post-deploy QA.

## A. Perceivable — color contrast (open follow-ups)

The 11 brand tokens were computed against the dominant cream background (`#FFFAEE`):

| Combo | Ratio | AA normal (4.5:1) | AA large (3:1) | Verdict |
|---|---|---|---|---|
| `text-jw-blue` (#1A2256) on `bg-jw-cream` (#FFFAEE) | 12.7:1 | ✅ | ✅ | Strong |
| `text-jw-ink` (#2A2D3A) on `bg-jw-cream` | 12.6:1 | ✅ | ✅ | Strong |
| `text-jw-muted` (#6B6860) on `bg-jw-cream` | 5.5:1 | ✅ | ✅ | Pass (just) |
| `text-jw-blue-soft` (#3B4A8A) on `bg-jw-cream` | 7.0:1 | ✅ | ✅ | Strong |
| `text-jw-coral` (#E8632B) on `bg-jw-cream` | **3.6:1** | ❌ | ✅ | **FAIL for normal text** |
| `text-white` on `bg-jw-coral` (#E8632B) | **3.5:1** | ❌ | ✅ | **FAIL for normal text** |
| `text-jw-cream` on `bg-jw-blue` (#1A2256) | 12.4:1 | ✅ | ✅ | Strong (footer/CTA OK) |

WCAG defines "large text" as ≥18pt (24px) regular or ≥14pt (18.67px) bold. In Tailwind terms:

- `text-2xl` (24px) and up: large at any weight.
- `text-xl` (20px) bold/semibold: borderline large.
- `text-lg` (18px) bold/semibold: technically still small (just under 14pt).
- `text-sm` (14px) and `text-base` (16px): always small, regardless of weight.

**Implication**: every `text-sm font-semibold text-jw-coral` link (and equivalents) currently sits below the 4.5:1 threshold. Direct grep counts ~38 such usages across the codebase (small-text coral plus small-text cream-on-coral). This is the single largest outstanding accessibility risk before launch.

### Coral-on-cream small text — affected files (sample)

- `components/beranda/petisi-preview.tsx:33,61` — "Lihat semua →" CTAs
- `components/beranda/janji-tracker.tsx:77`
- `app/page.tsx:136` — section "Lihat semua" link
- `app/karya/page.tsx:134`, `app/komunitas/page.tsx` (Reset filter)
- `app/main/tebak-kata/page.tsx:21`, `app/main/tebak-pejabat/page.tsx:34,72`
- `app/komunitas/[id]/ringkas-nala-button.tsx:23` (text-xs!)
- `app/tagih/page.tsx:183`, `app/tagih/[id]/page.tsx:71`, `app/tagih/baru/page.tsx:40`
- `app/aksi/petisi/[id]/page.tsx:50`, `app/aksi/polling/[id]/page.tsx:50`
- `app/kelas/[id]/modul/[modulId]/module-nav.tsx:16,31`
- `app/(auth)/daftar/signup-form.tsx:147` and `app/(auth)/masuk/login-form.tsx:260` — error messages (`text-xs text-jw-coral`)
- ~12 more `font-hand text-jw-coral text-sm` decorative kicker spans (lower risk — usually accompanied by an adjacent heading; can be marked `aria-hidden="true"` since they duplicate the section title)

### White-on-coral small CTA — affected files (sample)

- `components/site-header.tsx:63` — "Daftar" CTA (`text-sm font-semibold`, 14px)
- `app/komunitas/page.tsx:113` — "+ Mulai thread" CTA
- `app/tagih/page.tsx:161` — "Submit janji baru" CTA
- `app/aksi/polling-featured-card.tsx:30` — "Vote sekarang" CTA
- `app/karya/page.tsx` upload button (currently disabled, lower priority)

### Recommendation (deferred — own follow-up spec)

Three viable paths, in order of preference:

1. **Darken `--color-jw-coral`** from `#E8632B` to roughly `#C44E1A` (target ratio 4.6:1 against cream and 4.7:1 inverse). One global token edit; all surfaces update at once. Risk: brand calibration shifts — needs Mas sign-off.
2. **Switch small-text coral links to `text-jw-blue`** (12.7:1, very safe). Keeps the brand palette intact but flattens the visual identity of "tertiary action" cues.
3. **Bump small CTAs to `text-base font-bold`** so they cross the 14pt-bold threshold, plus underline on idle (not just hover). Cheapest visually but increases visual weight of every "Lihat semua" link.

**Acceptance per spec § 8** asks "audit + replace coral text usage with blue for small text". This audit documents the systemic finding rather than applying ~38 spot edits, which would risk visual regression across the entire surface and conflict with Window C's mobile responsive sweep. Recommend **a dedicated spec in Sprint 4** to apply option 1 (single token change) under brand review.

The existing AA test suite (axe-core in jsdom) does not catch these because color-contrast is disabled — see Methodology Disclosure. Manual contrast spot-checking with `https://contrast-ratio.com/` is the verification mechanism until headless-browser tooling is wired up.

## B. Operable

### B.1 Skip link — FIXED

`components/skip-link.tsx` (new). Renders `<a href="#main-content">Lompat ke konten utama</a>`, hidden via `sr-only`, becomes visually positioned at top-left when focused via Tab from URL bar. Wired in `layout.tsx` immediately after `<body>`. The wrapper `<div id="main-content" className="flex-1">` is the landing target; pages that have their own `<main>` (Beranda, Komunitas, Karya, Kelas, Aksi, Tagih, Main after this audit) sit inside it.

### B.2 Keyboard accessibility

All interactive surfaces are reachable via Tab:
- Site-header logo, nav links, profile/Masuk/Daftar CTAs, Nala trigger — all native `<a>` and `<button>`.
- Vote arrows on threads — `<button>` (not divs with onClick). `aria-pressed` reflects state. Now also has `focus-visible:outline` ring.
- Komunitas filter/sidebar, Karya tabs, Kelas filters, Janji filters — all `<button>` chips.
- Polling form, petisi sign button — `<button type="submit">`.
- Nala panel — focus is captured on open, trapped via Tab/Shift+Tab cycle, restored to last focused element on close. Esc dismisses (already implemented in `nala-panel.tsx:50–73`).
- Game tiles on `/main` — `<Link>` cards, fully keyboard activatable.

**No keyboard traps detected.** No element relies on `mouseover`/`mouseenter` exclusively.

### B.3 Focus indicator

Tailwind 4 default `focus-visible` ring is preserved; the SkipLink itself adds an explicit `outline-2 outline-offset-2 outline-jw-coral`. Vote arrows now add a ring on focus. The remaining buttons inherit Tailwind 4's default ring which uses the `--color-jw-blue-soft`-adjacent token via the browser's accent color, which appears reliably under default user agent styles.

**Open**: there is no project-wide `focus-visible` style override in `globals.css`. Default browser focus rings are visible enough for AA; if Mas wants brand-coloured rings as a polish task, that's a Sprint 4 follow-up (`*:focus-visible { outline: 2px solid var(--color-jw-coral); outline-offset: 2px; }`).

### B.4 Touch target ≥ 44×44px

Spec #20 (Window C, parallel batch 2) covers this in detail. Where overlap exists in this audit's scope: vote-arrow buttons currently render with `p-1` plus icon size 20 → roughly 28×28px hit area. **Recommendation**: bump padding to `p-2` (≈36×36px) or wrap in a 44px hit-target region. Defer to Window C / Spec #20 to avoid double-edit conflicts. **Tracked** but not applied here.

## C. Understandable

### C.1 Page title — pass

Each page exports `metadata.title` (e.g. `'Komunitas — Jubir Warga'`). Beranda uses the layout's default template `'Jubir Warga — Suara warga, rumahnya di sini.'`.

### C.2 HTML lang — pass

`<html lang="id">` already present in `layout.tsx:92` (pre-existing — no change needed).

### C.3 Form label association

Auth forms (`login-form.tsx`, `signup-form.tsx`) use `<label htmlFor="email">` paired with `<input id="email">`. Reply form (`komunitas/[id]/reply-form.tsx`) uses `<label>` wrapping the `<textarea>` (implicit association — also valid per HTML spec). Polling form (`aksi/polling/[id]/polling-form.tsx`) uses `<fieldset>` + `<legend>` + radio inputs with explicit labels.

**No outstanding form-label issues found.** Error messages live in adjacent siblings with `text-jw-coral` (color-contrast caveat above).

### C.4 Error messaging

Server-action error states (auth, vote-thread, sign-petisi) bubble back through `useFormState` and render in coloured spans adjacent to the failing field. **Recommendation** (defer): wrap error spans with `aria-live="polite"` and `role="alert"` so screen readers announce them on appearance. Currently they are visually present but not announced.

### C.5 Consistent navigation

Every page that renders `<SiteHeader>` does so with the same set of nav links in the same order (`Komunitas / Karya / Kelas / Tagih / Aksi / Main`). However **only Beranda actually renders `<SiteHeader>`** — the inner pages (Komunitas, Karya, Kelas, Aksi, Tagih, Main) currently render directly inside `<main>` without a header. This is a pre-existing structural issue, not introduced by this audit, and should be addressed in a follow-up that either:

- Moves `<SiteHeader>` into `app/layout.tsx` (cleanest, but layout becomes async to fetch user), or
- Adds `<SiteHeader>` to each page (requires user prop threading via `getUser()` per page).

This is **out of scope** for Spec #21 — flagged here for tracking. Manual Tab-from-URL-bar test will currently land on `<SkipLink>` then directly on the page's first interactive element, with no header to skip past. SkipLink still functions correctly (it jumps to `#main-content`), just less consequential without a header above it.

## D. Robust

### D.1 ARIA landmarks

After this audit:

| Landmark | Location | Status |
|---|---|---|
| `<header role="banner">` | `components/site-header.tsx` (when rendered) | ✅ explicit role added |
| `<nav aria-label="Navigasi utama">` | inside site-header | ✅ explicit label added |
| `<main id="main-content">` | per-page (Beranda inline, others now wrapped after this audit) | ✅ |
| `<footer role="contentinfo" aria-label="Informasi situs">` | `components/site-footer.tsx` | ✅ explicit role added |
| `<aside>` | komunitas-sidebar, karya top-kreator-sidebar | ✅ pre-existing |

Layout's outer wrapper is `<div id="main-content" className="flex-1">` (not `<main>`) to avoid nested-main violations on pages that already have their own `<main>` element. This is the same approach used by e.g. the BBC and GOV.UK design systems.

### D.2 Heading hierarchy

axe-core's `heading-order` rule fires for any skip from `h1` to `h3`, etc. One violation found and fixed:

- `app/aksi/polling-featured-card.tsx` was `<h3>` after the page-level `<h1>Aksi</h1>`. Fixed to `<h2>` since polling and the petisi `<h2>` section are siblings.

Other pages were spot-checked: Beranda (`h1` hero → `h2` section heads via `SectionHead`), Komunitas (`h1` page → `h3` thread title — defensible since each `ThreadRow` is a self-contained `<article>` and h-order rules within an article reset). Karya, Kelas, Tagih, Main: `h1` page → `h2` section → `h3` cards. All passing.

### D.3 ARIA live regions

- Vote-arrow score now has `aria-live="polite" aria-atomic="true"` so screen readers announce vote count changes.
- Toast/success messages on auth and reply-form: not currently in `aria-live` containers. **Recommendation** (defer): add `role="status"` + `aria-live="polite"` to the success banners.

### D.4 Modal/dialog ARIA

`components/nala/nala-panel.tsx` already ships:

- `role="dialog"` + `aria-modal="true"` + `aria-label="Chat dengan Nala"`
- `aria-hidden={!isPanelOpen}` so the off-screen panel is invisible to AT
- Focus trap: focus moves to close button on open, traps inside via Tab cycle, restores to triggering element on close
- Esc dismisses (`onKeyDown` listener)
- Backdrop `aria-hidden="true"` so it's not announced

**No fix needed.** This is a strong baseline.
